# 人格与长期一致 — 优化路线图（综合版）

本文档合并了两部分来源的优化建议，形成统一实施路线：

- **来源 A**：人格增强方向（情绪状态、关系记忆、主动性、口吻稳定、内心/表面、短期目标、拒绝与边界）
- **来源 B**：四大子系统（Reflection 反思坍缩、Relationship 羁绊状态机、Agent Loop 赛博生物钟、Emotion+ToM 情绪与心智）

---

## 一、优化目标与对应能力

| 目标 | 要解决的问题 | 对应能力/模块 |
|------|--------------|----------------|
| **长期一致** | 对话多了只记得碎片的“树木”，没有“森林” | Memory + Reflection（对话坍缩为高阶语义记忆） |
| **关系演变** | 永远“只如初见”，态度不随相处变化 | Persona + Relationship（羁绊值 + 动态 prompt） |
| **有活感** | 只会被动接话，没有“想找你说话” | Goal/Intention + Agent Loop（定时意图 + 主动行为） |
| **有温度** | 只记事实不记情绪，不懂潜台词 | Emotion 状态 + ToM 提示（情绪衰减 + 心智推断） |
| **口吻稳定** | 人格漂移、偶尔崩人设 | Task/示例 + 口头禅 + 拒绝边界 |

---

## 二、按模块的综合优化项

### 2.1 记忆层：Mem0 + Reflection（长期一致性子系统）

- **现状**：Mem0 存零散事实，metadata 含情绪/重要度/时间；chat_history 只做短期窗口。
- **优化**：
  - **Reflection 反思**：定时（如每日）按 `session_id` 读取该会话短期对话，用 Gemini 做总结（用户核心痛点、近期目标、对角色态度的变化等），以**高 importance** 写回 Mem0，并带该 session 的 `user_id`，便于后续检索。
  - **不要求**“挂机 4 小时”等复杂触发，首版仅定时即可。
  - **新增**：`src/brain/reflection.py`（读历史 → 调用 LLM 总结 → 写 Mem0）。

**与“关系记忆”的衔接**：Reflection 总结里可包含“和用户的关系进展”，输出可带 `intimacy_delta` 供关系子系统使用。

---

### 2.2 状态层：Relationship + Emotion（关系与表达 + 情绪连贯）

- **现状**：无持久关系/情绪状态，每轮相对独立。
- **优化**：
  - **关系**
    - **单一数据源**：`data/state/relationships.json`，按 `user_id`（如 `tg_123`）存 `intimacy`（0–100）。
    - **隐式更新**：每轮有效对话（用户发消息且得到回复）`intimacy += 0.1`（或 0.05），封顶 100。
    - **显式更新**（可选）：Reflection 阶段 LLM 输出 `intimacy_delta`（如 ±1～3），回写该表。
    - **Prompt 映射**：在 `prompt_assembler` 中按 intimacy 分 2–3 档注入一句关系描述，例如：
      - 低（0–30）：保持防备、礼貌疏离；
      - 中（31–70）：自然、可开玩笑；
      - 高（71–100）：卸防、口语化、可展现脆弱与依赖。
  - **情绪**
    - 在**同一状态模块**中维护 `current_emotion`（及上一轮情绪）。
    - **衰减/转移规则**：如“极度愤怒”下一轮只能为“余怒未消/冷漠/疲惫”等，禁止直接跳“开心”；过 N 轮或时间后再允许恢复中性。
    - 每轮可将当前情绪写入 Mem0 metadata，并可在 §6 或单独一句注入“当前情绪”。

- **新增**：`data/state/relationships.json`、`src/core/state_manager.py`（统一读写在手；Reflection 可调 state_manager 写 intimacy_delta）。

---

### 2.3 提示词层：prompt_assembler 的注入点

- **现状**：§1–§8 已固定；数据来自 prompts、persona、mem0、历史、vision、当前输入、task。
- **优化**（均在 `prompt_assembler` 内完成，不破坏 §1–§8 顺序）：
  1. **§2 角色卡之后 或 §6 之前**  
     - 根据 `state_manager` 的 **intimacy** 档位，插入**一段关系描述**（如“当前你与用户关系较亲密，可以更随意、流露依赖”）。
  2. **§6 环境感知**  
     - 在现有“当前时间”“本回合有图/无图”之外，增加**当前情绪**一句（来自 state_manager 的 `current_emotion`）。
  3. **§7 用户当前回合之前 或 task 之前**  
     - **ToM（心智理论）**：加一句 system——“回答前先在内心推断用户这句话背后的需求或情绪，再用人设口吻回应。”**不要求**模型输出“(ToM心智推测：...)”明文，避免解析与格式问题。
  4. **Task / 示例**  
     - **口吻稳定**：在 task 或 persona 示例中补充口头禅、禁用表述（如“作为 AI”“我无法”）、句式偏好。
  5. **拒绝与边界**  
     - 在 jailbreak/task 中说明：角色可按人设拒绝、敷衍或岔开话题（如傲娇式拒绝），不必有求必应。

**与“短期目标”的衔接**：若后续做“角色当前惦记的事”，可在 §2 或 §6 用同一 state 源注入一句（如“本会话里你一直想和用户说 XX”）。

---

### 2.4 主动性与活感：Agent Loop（赛博生物钟）

- **现状**：仅有“心跳看图”触发的主动说话，无基于时间/状态的意图生成。
- **优化**：
  - **定时唤醒**：如每 2–6 小时随机触发一次，不依赖“有用户输入”。
  - **Appraisal**：读取简单状态（如 `social_battery`：每轮对话 -1，每小时 +0.5，0–100 封顶）及可选“最近摘要”。
  - **Intention**：对**后端**做一次 LLM 推理（Internal Monologue），输入“闲置时长、电量、可选工具反馈”，输出**动作选择**（如：主动给用户发一条消息 / 什么都不做）。
  - **执行**：若选择“主动发消息”，则走现有对话流（插入一条占位或 system，生成一条回复并推送到当前会话）。

**首版可简化**：不接小红书等多平台工具，仅“定时 + 内部推理 + 可选主动发一条消息”；`social_battery` 用简单公式即可。

- **新增**：`src/core/agent_loop.py`（或等价模块），由定时任务/现有进程调起。

---

### 2.5 陪伴感与心智：Emotion + ToM（已在 2.2、2.3 覆盖）

- **情绪**：见 2.2（state_manager 维护、衰减规则、§6 注入）。
- **ToM**：见 2.3（仅 system 内一句“内心先推断再回应”，不要求可见的 ToM 输出格式）。

## 三、实施优先级与依赖关系

```
P0（先做，无依赖）
├── state_manager.py + data/state/relationships.json
├── 羁绊隐式更新（每轮 +0.1）+ prompt_assembler 按 intimacy 注入关系描述
└── 情绪 state + 简单衰减规则 + §6 注入当前情绪

P1（依赖 P0 / 现有 memory）
├── reflection.py：按 session 总结对话 → 高 importance 写 Mem0
├── 定时触发 Reflection（如每日）
└── ToM 一句注入（不要求输出格式）

P2（依赖 P0 状态 + 现有对话流）
├── agent_loop.py：定时唤醒 + 内部 LLM 意图 + 可选“主动发一条消息”
├── social_battery 简单公式
└── Reflection 输出 intimacy_delta 回写 relationships（可选）
```

## 四、与“之前优化建议”的对应关系

| 之前建议 | 归属到综合方案中的位置 |
|----------|------------------------|
| 情绪状态、当前心情 | 2.2 情绪 state + §6 注入；2.5 Emotion |
| 关系记忆、“记得你” | 2.2 Relationship + Mem0/Reflection 中的关系总结 |
| 主动性、先开口 | 2.4 Agent Loop + 现有心跳看图 |
| 口吻稳定、口头禅、禁用句 | 2.3 Task/示例 + 拒绝与边界 |
| 内心戏 vs 表面话 | 现有 task 已有人称与内心独白；可再强化“内心先推断”即 ToM |
| 短期目标、“惦记的事” | 2.3 中“可选：state 注入一句当前惦记”；Agent Loop 可生成该状态 |
| 拒绝与边界 | 2.3 jailbreak/task 说明 |

## 五、小结：如何优化（一句话版）

1. **状态**：用 `state_manager` + `data/state/` 统一管**羁绊值**和**当前情绪**，并做简单更新与衰减规则。  
2. **记忆**：用 **Reflection** 把短期对话坍缩成高阶语义记忆写回 Mem0，并按 session 隔离。  
3. **提示词**：在 **prompt_assembler** 里按 intimacy 注入关系、按情绪注入当前心情、加一句 ToM 指引，并在 task/示例里稳住口吻与边界。  
4. **主动**：用 **Agent Loop** 定时做意图推理，可选地触发“主动发一条消息”，让数字生命有“想找你说话”的感觉。  

按 **P0 → P1 → P2** 的顺序做，即可把“之前的优化点”和“四个子系统”统一落地。

## 六、近期执行清单（面试导向）

> 目标：把项目从“功能可用”提升到“可证明效果 + 可讲工程权衡”。

### 6.1 优先级结论

1. **先做评测与优化闭环（P0）**
2. **再做 Skill 调用能力（P1）**
3. **最后做额外功能扩展（P2）**

### 6.2 P0：评测与优化闭环（本周必须完成）

- 新建 `eval/` 目录与基准数据（建议 50~100 条）：
  - 角色一致性
  - 时间感知准确（“现在几点”类）
  - Mem0 命中率（是否引用正确历史事实）
  - 回复风格稳定（口吻/边界）
- 新建评测脚本（可离线跑）：
  - 输入固定测试集
  - 输出每项得分与总分
  - 保存结果到 `eval/results/*.json`
- 每次改动后记录“前后分数对比”，在 README/文档里保留至少 2 组优化案例。

### 6.3 P1：增加 Skill 调用能力（建议下周）

> 这是有价值的，但前提是有评测闭环来证明“加了 skill 后真的更好”。

- 目标：让主脑在需要时可调用内部 skill（工具能力）完成任务，而不是只靠纯文本生成。
- 最小落地方案：
  - 在 `src/brain/tools_registry.py` 定义 Skill 接口（输入/输出 schema 固定）
  - 在 `conscious` 调用链中接入“可选工具调用”流程（先推理是否调用，再执行，再回填结果）
  - 先做 2~3 个高价值 skill：
    - 时间/时区信息查询（统一与 `app_timezone`）
    - 结构化记忆查询（按 user_id、时间范围）
    - 简单任务型工具（如待办抽取或摘要）
- 验收标准：
  - 评测集中“工具型问题”正确率提升
  - 无工具问题不退化（延迟与风格保持）

### 6.4 P2：工程化补齐（并行推进）

- 测试：补单测/集成测试（`prompt_assembler`、`memory`、`llm_gateway`、`/api/chat`）
- 观测：Langfuse trace 字段统一（session_id、user_id、trace_name、模型、耗时）
- 稳定性：失败降级清单（网关超时、Mem0 不可用、Langfuse 异常）
- 交付：Docker + CI（lint/test）+ 一键运行说明

### 6.5 两周执行计划（建议）

- **第 1 周**
  - 完成 `eval/` 数据集与脚本
  - 跑第一版基线分数
  - 完成 1~2 项优化并产出分数提升报告
- **第 2 周**
  - 接入 Skill 调用最小闭环（2~3 个 skill）
  - 在评测集中新增工具类用例并验证提升
  - 补齐测试与 CI，形成可面试展示版本
